# AR バグハンター — サーバー・クライアント連携 実装プロンプト

> 用途: このドキュメントを GitHub Copilot に渡すだけで、**スタンドアロンモード（iOS 単体）** と **プロジェクター（大画面サーバー）＋ iOS クライアントの MultipeerConnectivity 完全同期連携** の両方を新規プロジェクトに実装できるようにする汎用テンプレートです。ファイル名・クラス名・変数名はプロジェクトごとに適切に命名してください。
> 対象環境: 最新の iOS SDK / 最新の Xcode / Swift / MultipeerConnectivity

---

## 1. 目的（Why）

本仕様の目的は以下を実現することです。

- **スタンドアロンモード**: iOS クライアント 1 台だけでエンティティ生成・捕獲を完結させる
- **プロジェクター連携モード**: プロジェクター側を唯一の権限サーバーとして扱い、iOS クライアントとの状態同期を一貫したフローで実現する
- スポーンや削除の最終決定権をサーバーに集約する（連携時）
- クライアントごとの差分や二重処理を防ぐ
- 途中参加・再接続・切断復帰を含めて、接続中クライアントの状態を整合させる
- 通信仕様と責務分離を明確にし、他プロジェクトでも流用しやすい形にする

---

## 2. 機能概要（What）

### 2-1. 動作モデル（モード別）

| モード | エンティティ生成権限 | 通信 | 表示 |
|---|---|---|---|
| スタンドアロン | iOS クライアント自身が自律生成 | なし（Multipeer セッション未起動） | AR 表示 |
| クライアント（連携） | プロジェクター（サーバー）のみが生成・ID 発行 | MultipeerConnectivity | AR 表示 |
| サーバー（プロジェクター） | プロジェクター（サーバー）のみが生成・ID 発行 | MultipeerConnectivity | 平面表示（非 AR モード） |

- **スタンドアロンモード**: iOS デバイスが唯一の生成権限を持つ。スポーンはローカルでのみ行われ、ネットワーク送受信は一切発生しない。
- **プロジェクター連携モード**: プロジェクター側（サーバー）が唯一の権限を持つ。エンティティの生成、安定 ID の発行、削除判定、全体同期はサーバーだけが行う。クライアントはサーバーから受け取ったイベントを AR シーンに反映するだけで、自律的に生成しない。

### 2-2. 通信フロー（クライアント ↔ サーバー）

- スポーン通知: Server → All Clients (`bugSpawned`)
- 削除要求: Client → Server (`bugRemoved`)
- 削除確定通知: Server → All Clients (`bugRemoved`)
- 得点通知: Server → Target Client (`bugCaptured`)
- ゲーム状態同期: Server → All Clients (`gameState`)
- 操作要求（発射/開始/リセット）: Client → Server (`launch` / `startGame` / `resetGame`)

### 2-3. 非対象

この文書では以下を扱わない。

- AR 空間への配置方法（ARKit アンカー計算の詳細）
- RealityKit / SpriteKit の描画詳細（マテリアル、アニメーション等）
- UI レイアウトやオーバーレイ表示
- ゲーム演出、移動アニメーション、スコア演出

## 3. 実装方針（How）

### 3-1. 接続方式（連携モードのみ）

- サーバー（プロジェクター）は `MCNearbyServiceAdvertiser` で広告し、`MCNearbyServiceBrowser` でも探索する
- クライアントは `MCNearbyServiceBrowser` でサーバーを発見したら即座に招待する
- サーバーは招待受理時に接続上限を確認し、空きがある場合のみ接続を許可する
- `MCSession` は暗号化必須の設定を使用する
- スタンドアロンモードでは Multipeer セッションを起動しない

### 3-2. 責務分離

- 通信層はメッセージの送受信、接続管理、再同期要求の処理だけを担当する
- ゲーム層は受信イベントを購読し、ローカル状態へ反映する
- 通信層は描画 API や UI API に依存しない
- スタンドアロンモードの生成ロジックは AR コーディネーター（AR セッション管理クラス）に閉じ込め、通信層を一切経由しない

### 3-3. 整合性維持（連携モード）

- サーバーはエンティティごとに `UUID().uuidString` を発行する
- クライアントは受信した安定 ID を削除まで保持する
- 途中参加時はサーバーが生存中エンティティ一覧を再送する
- 同一 ID の重複追加はクライアント側で無視する

### 3-4. 表示モード

- **プロジェクター側（サーバー）**: `ARView(cameraMode: .nonAR)` + `SKView` による平面 2D 表示。ARKit トラッキングは使用しない。固定カメラ（`PerspectiveCameraComponent`）で 3D エンティティを描画する。
- **iOS クライアント側（スタンドアロン / 連携）**: `ARView`（ARKit + RealityKit）による AR 表示。ARKit ワールドトラッキング + `ARAnchor` でエンティティを空間に固定する。

## 4. デバイスロール

### 4-1. プロジェクター（サーバー）

- **表示**: `ARView(cameraMode: .nonAR)` + 透明 `SKView` オーバーレイによる平面 2D 表示。ARKit トラッキング不使用。
- MultipeerConnectivity セッションの主導権を持つ
- サービス識別子はプロジェクト共通の文字列で固定する（例: `"<your-app>-game"`）
- 最大同時接続数は 3 クライアントとする
- 接続ごとにプレイヤースロットを 0, 1, 2 の順で割り当てる
- 接続直後に生存エンティティ一覧を新規クライアントへ送信して初期同期を行う
- サーバー側の生成コーディネーターがエンティティを自律スポーンし、スポーン通知コールバック経由で Multipeer ブロードキャストを呼び出す
- クライアントからの削除要求（`bugRemoved`）を受信したら妥当性を確認し、削除確定後に全クライアントへ中継通知する

### 4-2. iOS クライアント（連携モード）

- **表示**: ARKit ワールドトラッキング + RealityKit (`ARView`) による AR 表示。エンティティは `ARAnchor` ベースの `AnchorEntity` に紐付けて空間に固定する。
- サーバーへ接続し、受信したスポーン通知をローカル AR シーンに反映する
- エンティティ捕獲時はサーバーへ削除要求（`bugRemoved`）を送信する（サーバーが全クライアントへ中継）
- 発射（`launch`）、開始（`startGame`）、リセット（`resetGame`）の入力イベントをサーバーへ送信する
- サーバー未確定のローカル状態を権威状態として扱わない
- 重複スポーンや未知 ID の削除通知を受けてもクラッシュしない

### 4-3. iOS スタンドアロン

- **表示**: iOS クライアントと同様の AR 表示（ARKit + RealityKit）。
- Multipeer セッションは起動しない
- AR コーディネーターが `ARKit ARAnchor` ベースでエンティティを自律スポーンする
- 捕獲はすべてローカルで完結する。削除通知メソッドは呼ばれるが、接続ピアがないため送信は no-op になる
- スコア・タイマーはすべて iOS 側で管理する

## 5. 接続管理仕様

### 5-1. セッション確立

- クライアントはサーバー発見後に `invitePeer(_:to:withContext:timeout:)` を呼ぶ
- サーバーは接続数が 3 未満の場合のみ招待を承諾する
- 接続確立時に空きスロットの最小値を割り当てる
- 切断時は割り当て済みスロットを解放する
- 再接続時はその時点の空きスロットを再割り当てする

### 5-2. スロット管理

- サーバーは `[MCPeerID: Int]` 形式のマップでピア→スロット対応を管理する
- サーバーは `Set<Int>` 形式で使用中スロット集合を管理する
- スロット番号はクライアント識別と個別通知に使用する
- クライアント数が上限を超える接続要求は拒否する

### 5-3. 再同期

- サーバーは `MCSessionState.connected` 直後に現在の生存エンティティ一覧を対象クライアントへ送信する
- 再同期は通常のスポーン通知メッセージを個別送信する方式で行う
- クライアントは通常受信と同一経路で処理する

## 6. メッセージプロトコル

### 6-1. エンベロープ

全メッセージは JSON で送受信し、共通エンベロープを持つ。

```json
{
    "type": "<MessageType>",
    "launchPayload": null,
    "gameStatePayload": null,
    "bugCapturedPayload": null,
    "bugSpawnedPayload": null,
    "bugRemovedPayload": null
}
```

- `type` でメッセージ種別を表す
- 使用しないペイロードは `null` にする
- デコード失敗メッセージは破棄する
- 未知の `type` は破棄する

### 6-2. メッセージ種別

| MessageType | 方向 | ペイロード | 用途 |
|---|---|---|---|
| `startGame` | Client → Server | なし | ゲーム開始要求 |
| `resetGame` | Client → Server | なし | ゲームリセット要求 |
| `launch` | Client → Server | `LaunchPayload` | 発射イベント送信 |
| `bugSpawned` | Server → All Clients | `BugSpawnedPayload` | スポーン通知 |
| `bugRemoved` | Client → Server / Server → All Clients | `BugRemovedPayload` | 削除要求または削除確定通知 |
| `bugCaptured` | Server → Target Client | `BugCapturedPayload` | 個別加点通知 |
| `gameState` | Server → All Clients | `GameStatePayload` | 全体状態同期 |

### 6-3. ペイロード定義

#### LaunchPayload

```text
angle     : Float
power     : Float
timestamp : Double
```

#### BugSpawnedPayload

```text
id          : String
bugType     : BugType
normalizedX : Float
normalizedY : Float
```

#### BugRemovedPayload

```text
id : String
```

#### BugCapturedPayload

```text
bugType     : BugType
playerIndex : Int
```

#### GameStatePayload

```text
state         : String
score         : Int
timeRemaining : Double
```

## 7. 同期ルール

### 7-1. スポーン同期（連携モード）

1. サーバー側の生成コーディネーターが新しいエンティティ ID（`UUID().uuidString`）を発行する
2. サーバーが内部の生存エンティティ管理マップへ登録する
3. サーバーが `bugSpawned` を全クライアントへブロードキャストする
4. クライアントは未登録 ID のみ AR 空間に `ARAnchor` として配置する

### 7-1b. スポーン（スタンドアロンモード）

1. AR コーディネーターがローカルでエンティティ種別をランダム決定する
2. ARKit `ARAnchor` を世界座標に追加する（`arView.session.add(anchor:)`）
3. `session(_:didAdd:)` デリゲートで 3D エンティティと SpriteKit プロキシノードを生成する
4. ゲームマネージャーのスポーン通知メソッドは呼ばれるが、接続ピアがないため no-op となる

### 7-2. 削除同期（連携モード）

1. クライアントがエンティティを捕獲し、サーバー割り当ての安定 ID を `bugRemoved` として送信する
2. サーバーがその ID の生存を確認する
3. サーバーが内部状態から削除する
4. サーバーが `bugRemoved` を全クライアントへブロードキャストする
5. 全クライアントは対象 ID をローカル AR シーンから削除する

### 7-3. 遅延参加

1. クライアントが接続完了する（`MCSessionState.connected`）
2. サーバーが生存エンティティ取得コールバックで現在生存中の全 ID を列挙する
3. 各 ID について `bugSpawned` を当該クライアントへ個別送信する
4. クライアントはサーバー ID → ローカル識別子マップで重複を検出し、既存 ID を重複追加しない

### 7-4. 同時表示上限

- サーバーとクライアントは同じ上限値を共有する
- 推奨上限は 4 体（プロジェクト固有の定数として定義すること）
- サーバーは上限到達中に新規スポーンを確定しない
- クライアントは上限超過のスポーン通知を受信しても安全に破棄する

### 7-5. スタンドアロン削除フロー

1. ネット（発射物）がエンティティに命中し、捕獲コールバックが呼ばれる
2. AR コーディネーターの捕獲処理メソッドが `ARAnchor` をセッションから除去する
3. スコアはシーン → シーンデリゲート経由でゲームマネージャーへ反映される
4. 削除通知メソッドは呼ばれるが、接続ピアがないため no-op となる（ネットワーク副作用なし）

### As-Is: 初回スポーン要件（現行実装の明文化）

> この As-Is セクションは「現在このリポジトリで動いている挙動」の説明。次の To-Be セクションは「他プロジェクトで不具合再発を防ぐための採用必須ルール」。

- サーバー（プロジェクター）は `ProjectorBug3DCoordinator.attach(...)` 呼び出し直後に自律スポーンループを開始する
- 現行実装の初回スポーンは「即時」ではなく、`startAutonomousSpawning()` 内の `scheduleNextAutonomousSpawn(after: 1.5)` により **1.5 秒後** に発生する
- クライアント接続時は `MCSessionState.connected` 直後に `sendCurrentBugs(to:)` を実行し、**その時点で生存中のバグのみ**を再送する
- 接続時点で生存バグが 0 体の場合、接続イベント自体は成功しても `bugSpawned` 再送は発生せず、次の自律スポーンまで待機となる

### To-Be: 厳格運用ルール（他プロジェクトへ適用する必須指示）

他プロジェクトで「サーバー側でバグがスポーンしない」事故を防ぐため、以下を **MUST** とする。

1. サーバーは `ProjectorBug3DCoordinator.attach(to:bugScene:)` で `startAutonomousSpawning()` を開始した直後に、初回バグを **即時スポーン** する（初回のみ遅延禁止）
2. サーバーはクライアント接続直後に生存バグ一覧を送信する前に、一覧が 0 件なら 1 体を **即時スポーンし** てから同期送信する
3. `sendCurrentBugs(to:)` 呼び出し前に「生存バグが最低 1 体存在する状態」を必ず作り、接続直後に最低 1 件の `bugSpawned` を受け取れるよう保証する
4. 上記の即時スポーンはサーバー権威で行い、クライアント側で補完スポーン（接続同期の穴埋め目的でクライアントが勝手に生成すること）をしない
5. 即時スポーンに失敗（例: アンカー生成失敗、座標検証失敗、描画ノード初期化失敗）した場合はサーバー側で **0.2 秒間隔・最大 10 回** の再試行を行い、成功まで同期送信を完了扱いにしない
6. 10 回失敗した場合は同期失敗として `critical` ログを残し、対象クライアントを切断して再接続時に再同期フローをやり直す

## 8. サーバー側状態管理（プロジェクター）

### 8-1. 必須管理情報

- 接続中ピア一覧（`MCSession.connectedPeers`）
- ピア → スロット番号マップ（`[MCPeerID: Int]` 形式）
- 使用中スロット集合（`Set<Int>` 形式）
- 現在生存中のエンティティ管理マップ（自律スポーン用 / 同期スポーン用それぞれ）
- エンティティ ID とスポーン情報（正規化座標・種別）の対応

### 8-2. 要件

- 生存中一覧は遅延参加同期用の「生存エンティティ取得コールバック」にそのまま利用できる形で保持する
- 削除確定前に一覧から消さない
- 接続イベントとゲームイベントの更新順序が競合しないよう、すべて同一スレッド（メインスレッド）で処理する
- 共有マップへのアクセスはロックまたは同等の排他制御で保護する

---

## 9. クライアント側状態管理

### 9-1. 必須管理情報（連携モード）

- サーバー接続状態フラグ
- サーバー発行 ID → ローカル識別子（`ARAnchor` の UUID 等）マップ
- ローカル識別子 → サーバー発行 ID 逆引きマップ
- ローカル識別子 → 3D エンティティ / プロキシノードのマップ

### 9-2. 必須管理情報（スタンドアロンモード）

- ローカル識別子（`ARAnchor` UUID）→ エンティティ種別マップ
- ローカル識別子 → 3D エンティティ / プロキシノードのマップ
- サーバー ID マップ群（サーバー ID → ローカル識別子 / 逆引き）は不使用のまま（空）

### 9-3. 要件

- 同一 ID のスポーン通知を重複適用しない（サーバー ID → ローカル識別子マップで存在確認）
- 未知 ID の削除通知を受けても無視する（マップに存在しない場合は早期リターン）
- セッション切断時は接続依存状態を安全に破棄または再接続待ちに戻す
- 通信コールバックから UI を直接触る場合はメインスレッドへ移譲する（`DispatchQueue.main.async`）
- RealityKit レンダースレッドから UIKit に直接アクセスしない（キャッシュ変数経由でアクセスする）

---

## 10. エラーハンドリング要件

- JSON エンコード失敗時はログを残して送信を中止する
- 接続先が 0 件なら送信をスキップする
- デコード失敗時はメッセージを破棄する
- 未知 ID の削除要求または削除通知はサイレントスキップする
- 再同期用コールバックやデータソースが未設定なら空振りで終了する
- 切断済みピアへの個別送信失敗はクラッシュ原因にしない

---

## 11. コーディング規約

- サーバー・クライアント連携ロジックのみを対象に変更する（**スタンドアロンの生成ロジックは変更しない**）
- 描画ロジックと通信ロジックを同一責務に混在させない
- `switch` は網羅的に書き、安易な `default` を使わない（ゲームモード・メッセージ型・エンティティ種別の各列挙型すべて）
- 新しい定数は型の内部に閉じ込め、マジックナンバーを避ける
- 共有状態はスレッドセーフに扱う（`NSLock` / `DispatchQueue.main` で保護）
- メッセージ型、サービス識別子、接続上限はサーバーとクライアントで必ず一致させる
- スタンドアロンモードのコードは通信層を経由させない

---

## 12. Copilot への実装指示

- サーバー・クライアント連携の変更だけを行う（**スタンドアロンは変更しない**）
- AR 表示、描画、UI、演出、ゲームバランスには触れない
- 既存コードのうち通信責務・エンティティ生成責務に直接関係しない部分は変更しない
- 必要なら通信層の型定義、送受信処理、接続管理、再同期処理のみを更新する（スタンドアロン生成ロジックは変更しない）
- 変更時は 1 つの PR 単位に収まる範囲で実装する

## 13. 実装ファイルに求められる責務

ファイル名・クラス名は各プロジェクトで自由に命名してください。以下は責務単位の分割指針です。

| 責務 | 配置先の指針 | 主なコンテンツ |
|---|---|---|
| メッセージ型定義 | Shared（双方が参照できる場所） | `MessageType` 列挙型、`GameMessage` エンベロープ、各ペイロード構造体、エンティティ種別列挙型 |
| クライアント側 Multipeer ラッパー | iOS Controller 層 | `MCSession` / `MCNearbyServiceAdvertiser` / `MCNearbyServiceBrowser` の管理、メッセージ送受信、デリゲート転送 |
