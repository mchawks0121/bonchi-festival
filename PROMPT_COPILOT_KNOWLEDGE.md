# ぼんち祭り バグハンター — GitHub Copilot ナレッジドキュメント生成プロンプト

> **用途**: このファイルを GitHub Copilot（チャット／エージェントモード）に渡すことで、  
> このプロジェクトを通じて得た GitHub Copilot の活用ナレッジ・プロンプトエンジニアリング・  
> ベストプラクティスをまとめた `COPILOT_KNOWLEDGE.md` を生成できます。  
> **対象**: GitHub Copilot を使い始めるチームメンバー・開発者  
> **最終更新**: 2025年4月

---

## このファイルの使い方

1. GitHub Copilot のチャット欄で **このファイル** と **このリポジトリ** を「Add Files」でコンテキストに追加する。
2. 以下の「▶ AI へのプロンプト」ブロックをそのままコピーして投げる。
3. 出力された Markdown を `COPILOT_KNOWLEDGE.md` として保存する。
4. チームの Wiki や README からリンクして共有する。

> **ポイント**: このプロンプトはリポジトリのコード・既存ドキュメント（`SPEC.md` / `PROMPT.md` /  
> `copilot-instructions.md` / `PROMPT_PM.md` / `PROMPT_26.md`）を読み取り、  
> **実際の開発経験から導いた具体的なノウハウ**を生成します。  
> 抽象的なテクニック集ではなく、このプロジェクト固有の実例を含めてください。

### 1. GitHub Copilot とは ― このプロジェクトでの活用概要

- Copilot の主要モード（インライン補完 / チャット / エージェント）の違いと使い分け
- このプロジェクトで実際に使ったモードとユースケース（表形式）
- エージェントモードとチャットモードの使い分け判断基準

### 2. プロンプトエンジニアリング基礎

- Why / What / How の3点セットが必須な理由（このプロジェクトの失敗談から導く）
- 良いプロンプトと悪いプロンプトの対比（Before / After 形式、実例を使う）
- コンテキスト設計: 何をどの順番で与えるか
  - ファイルの選び方（`SPEC.md` / `PROMPT.md` をピン留めする理由）
  - コンテキスト量の最適化（多すぎる / 少なすぎる問題）
- 制約の書き方: 禁止事項・使用フレームワーク・スレッドモデルを明示する効果

### 3. このプロジェクト固有のプロンプトテンプレート集

以下の用途ごとに、実際に使えるプロンプトテンプレートをコードブロックで提示する。

3-1. 新規ファイル実装依頼テンプレート

    - Swift 6 / RealityKit / MultipeerConnectivity 向けの制約セット

3-2. バグ修正・デバッグ依頼テンプレート

    - スレッドエラー（@MainActor 違反 / actor isolation）の調査依頼例

3-3. コードレビュー依頼テンプレート
    - Swift 6 strict concurrency 観点のレビュー依頼例
    - セキュリティ（MCSession 暗号化 / 強制アンラップ禁止）確認依頼例

3-4. リファクタリング依頼テンプレート

3-5. ドキュメント生成依頼テンプレート

### 4. copilot-instructions.md の設計ノウハウ

- `.github/copilot-instructions.md` がどのように機能するか（仕組みの説明）
- このプロジェクトで実際に記述したルールとその効果（表形式）
  - コーディング規約（switch exhaustive / 強制アンラップ禁止 等）
  - ドキュメント更新ルール（SPEC.md / PROMPT.md の同時更新）

### 5. PROMPT.md / SPEC.md パターン ― AI の記憶を補う設計

- なぜ AI は「記憶」を持てないのか（ステートレスな LLM の特性）
- PROMPT.md（各ファイルの実装説明）をコンテキストとして使う効果
- SPEC.md（仕様書）を生きたドキュメントとして維持する運用ルール
- コードと同 PR でドキュメントを更新させる仕組み（copilot-instructions との連携）
- AI に「前回と同じ品質を再現させる」テクニック

### 6. エージェントモード活用術

- エージェントモードと通常チャットの違い（自律的ファイル編集の仕組み）
- このプロジェクトでのエージェントモード活用シーン（実例）
  - 複数ファイルにまたがるリファクタリング
  - フレームワーク移行（SceneKit → RealityKit 等）
  - ドキュメント自動生成
- エージェントに指示する際の注意点

  - レビューポイントをあらかじめ指定する

### 7. チーム開発での Copilot 活用ルール

- Copilot の出力を「そのままマージしない」ためのレビュー文化の作り方
- マージ前チェックリスト（このプロジェクトで実際に運用したもの）
  - Swift 6 strict concurrency の確認点
  - セキュリティチェック項目
  - ドキュメント更新の確認
- チーム内でのプロンプト共有方法（PROMPT_26.md のような「プロンプト集」の運用）
- 初心者メンバーへの Copilot 導入のコツ
  - 最初に教えるべき3つのこと
  - 避けさせるべき失敗パターン

### 8. iOS / Swift 開発固有のプロンプトテクニック

- Swift 6 strict concurrency エラーを Copilot に直させるコツ
  - actor isolation エラーの依頼例
  - Sendable 準拠の追加依頼例
  - @MainActor / nonisolated の使い分け依頼例
- ARKit / RealityKit 向けのプロンプト工夫
  - RealityKit の Entity / AnchorEntity パターンを前提とした指示方法
  - レンダースレッド問題（UIKit 直アクセス禁止）を Copilot に守らせる方法
- MultipeerConnectivity のセキュリティ要件を守らせる方法
  - encryptionPreference: .required を必ず使わせるプロンプト例
- フレームワーク移行（旧 API → 新 API）の効率的な依頼方法

### 9. プロンプトエンジニアリング アンチパターン

このプロジェクトで経験した「やってはいけない」プロンプトパターンを具体例とともに列挙する。

- ❌ 曖昧な依頼（「バグを直して」「いい感じにして」）
- ❌ コンテキスト不足（関連ファイルを渡さずに依頼）
- ❌ スコープが広すぎる依頼（複数機能を一度に依頼）
- ❌ 制約を後から追加する（最初のプロンプトで制約を漏らす）
- ❌ AI の出力を確認せずに承認する
- ❌ フレームワーク指定を省略する（SceneKit か RealityKit か曖昧）
- ❌ switch の default 禁止を伝え忘れる

各パターンについて: 悪い例（Before）→ 良い例（After）→ 改善のポイント の形式で記述する。

### 10. 継続開発のための Copilot 活用フロー

新機能追加・バグ修正・リファクタリングそれぞれの推奨フロー図（テキストベース）と

10-1. 新機能追加フロー

10-2. バグ修正フロー
  エラー収集 → Copilot にデバッグ仮説を出させる → 修正依頼
  → テスト追加依頼 → レビュー

10-3. リファクタリングフロー
  対象の特定 → Copilot に移行計画を出させる → ファイル単位で依頼
  → ドキュメント更新
