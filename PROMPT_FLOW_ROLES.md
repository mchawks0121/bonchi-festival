# ぼんち祭り バグハンター — AI 駆動開発フロー & 5人役割分担プロンプト集

> **用途**: このファイルのプロンプトを GitHub Copilot（チャット／エージェントモード）に渡すことで、
> AI 駆動開発の全体フローと、5人チームの役割分担表を自動生成できます。
> **バージョン**: Xcode 26 / iOS 26 / Swift 6.0
> **最終更新**: 2025年5月
>
> **チームスキル前提**:
> - ロール 1（テクニカルリード）: GitHub Copilot・iOS 開発の経験あり
> - ロール 2〜5（4人）: **AI 駆動開発は初めて**。Copilot の使い方から丁寧にサポートが必要

---

## このファイルの使い方

1. GitHub Copilot のチャット欄でこのファイルと **このリポジトリ** を「Add Files」でコンテキストに追加する。
2. 目的に応じて以下の「▶ AI へのプロンプト」ブロックをコピーして投げる。
3. 出力された Markdown を該当ドキュメントに貼り付けて調整する。

### 1. AI 駆動開発とは（このプロジェクトでの定義）

- AI（GitHub Copilot）を「補助ツール」ではなく「開発ドライバー」として使う姿勢
- 人間の役割：要件定義・レビュー・統合テスト・最終判断
- AI の役割：実装・リファクタリング・ドキュメント更新・テストコード生成
- このプロジェクトの文脈でなぜ AI 駆動が有効か（チーム規模・技術スタックの複雑さ）
- **初心者メンバー向け一言説明**:
  「Copilot は優秀なペアプログラマー。コードを書かせて、自分はレビューと判断に集中する」
  「最初は完全に理解できなくてもOK。レビューチェックリストに沿って確認するだけでいい」

### 2. 全体フロー図（テキスト図または表形式）

以下のフェーズを網羅したフロー図を ASCII / Markdown で生成する:

  Phase 0: 環境構築・共有知識の整備
    └─ Xcode 26 インストール、copilot-instructions.md 設定、SPEC.md / PROMPT.md ピン留め
    └─ 【初心者向け】GitHub Copilot の有効化・ライセンス確認手順（スクリーンショット付きで説明）

  Phase 0.5: 初心者オンボーディング（リーダーが実施）★新フェーズ
    └─ Copilot の 3 モード体験ワークショップ（30分）
        - インライン補完: 関数名を打ち始めて Tab で受け入れる体験
        - チャットモード: 「このコードは何をしていますか？」と質問する体験
        - エージェントモード: 「〇〇ファイルを作って」と指示する体験
    └─ Why / What / How テンプレートを使って最初のプロンプトを全員で作成する演習
    └─ レビューチェックリストを使ったコードレビュー練習（既存コードを素材に）
    └─ 【完了基準】全員が「プロンプト投入 → 出力確認 → チェックリストでレビュー」を1回体験できた

  Phase 1: タスク設計（人間が担う）
    └─ 1PR = 1機能 の粒度でタスクを定義
    └─ Why / What / How の 3 点セットをプロンプトとして起票

  Phase 2: AI 実装サイクル（1タスクあたり）
    └─ プロンプト投入 → Copilot エージェント出力 → 人間レビュー → 修正 → マージ

  Phase 3: ドキュメント自動同期
    └─ コード変更時に SPEC.md / PROMPT.md を同一 PR で Copilot が更新

  Phase 4: 統合テスト・QA
    └─ Swift Testing スイート生成 → 実機テスト → 不具合 Issue 起票 → 修正ループ

### 3. 1タスク実装の標準サイクル（ステップ詳細）

以下のステップを番号付きで詳述する:

  Step 1: タスク定義（人間）
    - Why / What / How の 3 点セットを書く
    - ブロッカー（前提タスク）の確認
    - ブランチ名決定（feat/<タスクID>-<ファイル名>）
    - 【初心者向け】PROMPT_26.md の該当プロンプト（P-XX）をコピーして使う。
      ゼロから書く必要はない。空欄を自分のタスクに合わせて修正するだけでOK。

  Step 2: コンテキスト設定（人間）
    - Copilot のチャットに SPEC.md / PROMPT.md / 関連ファイルを Add Files で追加
    - copilot-instructions.md が最新かを確認
    - 【初心者向け】Add Files の操作手順:
        1. Copilot チャット欄の 📎 アイコンをクリック
        2. SPEC.md を選択 → もう一度 📎 → PROMPT.md を選択
        3. 「Files added」と表示されたらOK

  Step 3: プロンプト投入（人間）
    - Why / What / How を含む構造化プロンプトを投げる
    - 「SceneKit 禁止」「nonisolated(unsafe) 禁止」等の禁止事項を必ず明記
    - 【初心者向け】禁止事項はテンプレートにすでに含まれているので削除しないこと。
      プロンプトが長くても問題ない。むしろ詳細なほど Copilot の精度が上がる。

  Step 4: AI 実装（Copilot エージェント）
    - コード生成・ファイル作成・SPEC.md / PROMPT.md 更新を自律実行

  Step 5: レビュー（人間）
    - switch 網羅性確認（default: がないか）
    - @MainActor / Sendable 準拠確認
    - nonisolated(unsafe) がないか
    - MCSession.encryptionPreference が .required か
    - 強制アンラップ（!）がないか
    - 【初心者向け】後述の「AIレビューチェックリスト」をそのままコピーして上から順に確認する。
      意味が分からない項目はリーダーに Slack/Discord で質問すること（即レスする）。

  Step 6: 修正ループ（AI ↔ 人間）
    - レビューコメントを次のプロンプトとして投げる
    - 最大 3 往復以内で収束させる目安
    - 【初心者向け】修正プロンプトの例:
        「Step 5 のレビューで以下の問題が見つかりました。修正してください: [コピペ]」
      これだけで Copilot が修正案を出す。自分でコードを書き直す必要はない。

  Step 7: マージ & ドキュメント確認（人間）
    - PR のタイトルが「<タスクID>: <何をしたか>」形式か確認
    - SPEC.md / PROMPT.md の更新が同一 PR に含まれているか確認

### 4. Copilot 3 モードの使い分けフローチャート

  変更ファイルが 2 個以上 ──────────→ エージェントモード
  フレームワーク移行・大規模リファクタ → エージェントモード
  コンパイルエラーを直したい ───────→ エージェントモード
  「なぜこうなるか」を理解したい ────→ チャットモード（@workspace）
  1 箇所の小さな修正 ───────────→ インライン補完 or チャット
  定型コード（switch 全ケース等） ───→ インライン補完（Tab 受け入れ）

  上記を視覚的なフローチャート（ASCII / Mermaid）で表現すること。

### 5. プロンプト必須 3 点セット テンプレート

  このプロジェクト専用の Why / What / How テンプレートを以下の形式で生成する:

  【Why】
  （目的・背景。例：「ぼんち祭り会場で参加者が直感的にスリングショット操作を体験できるようにするため」）

  【What】
  対象ファイル: （例: Controller/SlingshotNode.swift）
  クラス名: （例: SlingshotNode）
  実装するプロパティ・メソッド:
    - （箇条書きで列挙）
  成果物:
    - （完成後の状態を箇条書きで）

  【How】
  フレームワーク制約:
    - 3D は RealityKit のみ。ARSCNView / SCNNode / SCNScene 等 SceneKit 系 API は使用禁止
    - SpriteKit 操作は main thread のみ
    - UIKit 操作は @MainActor 隔離
  Swift 6 制約:
    - nonisolated(unsafe) 使用禁止
    - BugType の switch は網羅的（default: 禁止）
    - 強制アンラップ（!）を本番コードに含めない
  セキュリティ制約:
    - MCSession.encryptionPreference は .required を使用
    - USDZ はアプリバンドルからのみ読み込む
  コメント要件:
    - 実装意図（なぜこの方法を選んだか）を必ずコメントに記載
    - セキュリティ上の考慮点をコメントに記載
    - 制約・注意点をコメントに記載

### 6. AI 生成コードのレビューチェックリスト

  マージ前に人間が確認する必須項目をチェックボックス形式で生成する:

  コード品質:
  □ switch が exhaustive か（BugType / GameState / GameMode 等）
  □ nonisolated(unsafe) が含まれていないか
  □ 強制アンラップ（!）が本番パスに含まれていないか
  □ guard let / if let で適切に nil チェックされているか

  スレッド安全性:
  □ RealityKit のエンティティ操作が @MainActor 隔離か
  □ SpriteKit の SKScene / SKNode 操作が main thread か
  □ NSLock のロック範囲が最小限か（デッドロックリスクなし）
  □ @Published プロパティへのアクセスが main thread か

  セキュリティ:
  □ MCSession.encryptionPreference が .required か
  □ 外部ファイルアクセスがアプリバンドル内に限定されているか

  ドキュメント:
  □ SPEC.md が同一 PR で更新されているか
  □ PROMPT.md が同一 PR で更新されているか
  □ 新規 public API にコメントが付いているか

  テスト:
  □ 単体テスト可能なロジックに Swift Testing テストが追加されているか
  □ 既存テストがすべてパスするか

### 6. AI 駆動開発における 5 人の協業ルール

  5人全員が守るべき協業ルールを箇条書きで生成する。
  **初心者メンバーが迷わず守れる**よう、ルールごとに「なぜそうするか（理由）」を一言添えること。

  コード生成:
  - AI が生成したコードは必ず別ロールのメンバーがレビューする（self-review 禁止）
    → 理由: 自分では気づけないミスを別の目で発見するため
  - プロンプトは Why / What / How の 3 点セットで書く（省略禁止）
    → 理由: 省略すると AI が文脈を誤解して的外れなコードを生成するから
  - ブランチ名: feat/<タスクID>-<ファイル名> （例: feat/p-06-bug3dnode）
    → 理由: 何のタスクかをブランチ名だけで識別できるようにするため
  - コミット: <タスクID>: <何をしたか> （例: P-06: Add Bug3DNode RealityKit entity）
    → 理由: 履歴を見たときに変更内容が一目でわかるようにするため

  初心者サポート:
  - 分からない Swift / Copilot の用語があったら、まず Copilot に「日本語で説明して」と聞く
  - それでも解決しない場合は GitHub Issues に `question` ラベルで起票する
  - リーダーは初心者メンバーの質問に **当日中に回答**する（SLA）
  - 週次定例でメンバー全員が「今週 Copilot で詰まったこと」を1つシェアする（ナレッジ共有）

  ドキュメント同期:
  - コードに変更を加えたら SPEC.md / PROMPT.md を同一 PR で更新する（QA & AI Champion が番人）
    → 理由: ドキュメントが古くなると次の Copilot プロンプトの精度が下がるから
  - SPEC.md の変更は必ずテクニカルリードが承認する

  コミュニケーション:
  - ブロッカーが発生したら当日中に GitHub Issues に起票する
  - 毎週の定例ミーティングでは「AI が生成したコードで詰まったポイント」を共有する
  - ナレッジは COPILOT_KNOWLEDGE.md に随時追記する（QA & AI Champion が管理）
